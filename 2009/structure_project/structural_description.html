Content models and reusable descriptions of structure

Sigfrid Lundberg
The Royal Library, Copenhagen

Abstract

This project will study methods for identification, indexing, search and retrieval of arbitrary parts or segments of structured information objects.

The ability to address arbitrary points in texts, images or videos is a basic requirement for using the resources in a scholarly context, but also for annotation and crowdsourced cataloging. Finally, without it, it is more or less impossible to participate in the web of data.

Most consumption environments are user interfaces for services provided by the library, but some such as RSS feeds, are for end users -- or possibly in his or her feed reader.

2. Standards and technologies

2.1 Modeling structure

In the area of mark-up languages, a content model is the rules for the configuration of element and/or data content allowable in instances of an element type. I.e., a content model is a range of objects that may be contained inside a container object. This means that each element has its own content model. A related usage of the term content model emanates from the partly overlapping communities around the Metadata Encoding and Transmission Standard (METS) and Fedora Commons. See the nice overview of of Fedora content models provided by, Indiana Digital Library and the set of type attribute values used within the Australian METS Profile. In this text, I will use the term content structure rather than content model, since I need to be able to discuss the content model of an element as well as the internal structure of physical as well as digital information objects.

2.2. Modeling text

An XML content model is the set of content objects that are valid inside a given element. A content structure (sensu a Fedora content model) is total valid tree of an information object, for instance an entire document. Fedora comes equipped with an object oriented data model capable of handling many different content structures.

There are a large number of mark-up languages available. All languages aimed for textual documents builds upon a common theory of text which is usually referred to as the Ordered Hierarchy of Content Objects (OHCO) model. The model is problematic for various reasons. The theory behind Fedora and METS is basically the same, so they come, I suppose, with the same problems.

2.3. Resource and content description

There are a few standards for the description, encoding and transmission of structural metadata. We may distinguish between descriptive encoding systems used within certain communities such as TEI Manuscript Description and Encoded Archival Description that are used to describe cultural heritage materials such manuscripts and archives, whereas tools such as DIDL can be used to describe the contents inside multimedia objects of various kinds.

2.4 Dissemination of structure

The dissemination and syndication of aggregates of resources is simplified by standards such as OAI-ORE), (such as Hierarchy Relations for Atom). These standards are complemented by vocabularies such as IANA's relation type vocabularies e.g., for use with atom:link tag (compare the corresponding vocabulary for html 4 and html 5 ).

3. Project

The project will address three hypotheses:

  1. The normalization of structural information is essential for making different media types available through a single user interface.
    • This project will study methods for identification, indexing, search and retrieval of arbitrary parts or segments of structured information objects.
  2. The normalization should be based on metadata schemes or text encoding formats, not applications, which embodies the data model.
    • This is in accordance with the philosophy that applications are like fish and data is like wine, where only the latter become more valuable as it ages.
    • The metadata schemes and text encoding formats should among the successful ones according to the principles outlined by Tim Bray in his article On Language Creation
  3. Items inside structured objects must be accessible through cool persistent http URIs
