Background

Laurentius is a catalogue of the Medieval manuscripts at Lund University Library. It contains a database for searching the content of this collection and also scanned images of a better part of it -- as of writing this (May 2005) 19748 images with the extension .tiff, each of which is available in high resolution (tiff), low resolution (jpeg) and small thumbnails (gif).

The manuscripts using a descriptive format -- Master (XML-based) which is thoroughly documented 1 The images are produced by a photographer down at the main library.

How to RUN Laurentius

The Laurentius services starts at system boot. It depends on two servers, an Apache web server and a fairly rusty version of Zebra (the free version, from before Indexdata made the commercial version). They start at the same run levels.

The two servers are started, as usual, with scripts in /etc/init.d namely apachectl and zebra-laurentius

How to UPDATE the Laurentius database?

Laurentius consists of two components -- the manuscript descriptions and the image database. The two are connected using a convention for the naming of images (see ./technical_documentation/image_naming_convention.text).

Updating the image database

Images arrive by ftp to the update directory. Bengt usually puts 'recto' and 'verso' images in seperate directories. These have to be renamed according to the naming conventions, and moved to the appropriate storage directory.

Deeper explanation (Mattias, 2003-06-12):

  1. All images must be renamed after the given naming-scheme (see separate entry and look at examples). There are a few 'rename'-scripts in the bin-dir that apply to different situations - you should really read through them to choose the right one, I'm afraid. Most of them let you do dry-runs of the renaming which might be helpful.
  2. The renamed tiff-originals must be duplicated as jpegs for online viewing, as well as gif-thumbnails for browsing of manuscripts. /home/scriptorium/bin/untiff is a tcsh-script that can work as an example. Please note that geometries might have to be altered, as the tiff-images size & geometry can differ between (and even within!) volumes...
  3. A few files need to be created/updated for each volume:
    • /home/scriptorium/storage/VOLUME/.images_per_row.txt Contains '1' or '2', 2 being the normal "one image per page" setup, 1 telling us that the images of this volume are of two pages each.
    • /home/scriptorium/storage/VOLUME/.images_in_order.txt All images that contain folios of any kind are listed in order
    • /home/scriptorium/storage/VOLUME/.bindings_in_order.txt - Any images of the volumes binding listed like this: 
      Mh_43b-ucext
Mh_43b-lcext
Mh_43b-ucint
Mh_43b-lcint
Mh_43b-spine
Mh_43b-foredge
Mh_43b-head
Mh_43b-tail
    • To simplify a consistent display of these images, please fill out any single blanks (only for volumes with separate images for recto & verso): 
      Mh_37-ucext
Mh_37-lcext
BLANK
Mh_37-lcint
Mh_37-spine
Mh_37-foredge
Mh_37-head
Mh_37-tail
    • /home/scriptorium/storage/VOLUME/.preliminaries_in_order.txt - Any images of the volumes preliminaries, in display-order Also here some "padding" might be needed to get the display "balanced" enough...: 
      BLANK
Mh_37-f_I_r
Mh_37-f_I_v
BLANK
    • /home/scriptorium/storage/VOLUME/.image_details.txt - A mapping: IMAGE\tIMAGE_DETAIL Maps all images that have details.
  4. A few scripts (/home/scriptorium/bin) should be run (in this order):
    • split-images.pl --volume=VOLUME -- This is only needed when .images_per_row.txt contains '1'.
    • create_volume_dbs.pl --volume=VOLUME -- This creates the basic db-files needed for each volume (image-order => image-name and vice-versa...)
    • create_volume_index.pl [OPTIONS] -- With the right options, this script will build all (x)html-pages needed for browsing a volume. It calls a few other scripts and does some XML/XSLT-work to extract text-data for display with the images. Build pages for specific volume: 
      		   create_volume_index.pl --volume=VOLUME --mapfiles --subsets
      Build pages for all volumes: 
      		  create_volume_index.pl --mapfiles --subsets

Updating the bibliographic records

Eva Nylander (and some of her co-workers) writes the records using some XML authoring tool. Sigfrid gets them as mail attachments. To put new or update records in involves the following

  • Validate the documents. For historical reasons, this is currently done as follows.
    1. This is a time-consuming and sometimes tedious task. Create an empty directory, copy the files into 
      		  /home/scriptorium/search/RECORDS/
      Note that we have the DTD among these files, and a file called all_records.xml (which merges all the records into one document).
    2. Unpack the new files and put them into the directory. First, Make sure that the new files are well formed XML. For example: 
      		  xmllint file.xml
      It happens occasionally that these files contain undefined entities. Most entities found in Laurentius are already defined in all_records.xml, so it is easier to xmllint all_records.xml than the individual Then start correcting!
    3. When the records are well-formed, then it is time to make them valid as well. Please note that validity is not necessary for the records to work but it does help. There are records that doesn't validate today. It is hard to give exact guidelines as to what kind of errors are acceptable and which are not.
  • Go to /home/scriptorium/search/RECORDS/ and check in and lock the current records using rcs, that is ci -l *xml. Some may not have change, som might have. Copy the new records into the directory.
  • Update the zebra database.
    1. ssh to hex
    2. Go to the search directory and perform the following tasks to rebuild various databases and to index the database: 
      		  cd /home/scriptorium/search/
		  bin/clear-items.sh
		  bin/item-indexer.pl
		  bin/mh_cleaner.pl
		  bin/mh_transformer.pl
		  zebra/run_zebraidx
    3. Rebuild the user interfaces and browsing structures: 
      		  cd /home/scriptorium/
		  bin/create_volume_index.pl --mapfiles --subsets

Imagefile naming conventions


Från "utsidan":
===============
* Upper cover exterior:	ucext
* Upper cover interior:	ucint
* Lower cover interior:	lcint
* Lower cover exterior:	lcext
* Foredge:	foredge
* Spine:	spine
* Head:	head
* Tail:	tail

Bladen, 'folios':
=================
* Vanliga sidor, recto & verso:	f_<NR>_(r|v)
* "Försättsblad" (preliminaries), recto & verso:	f_<RNR>_(r|v)
[preliminaries namnges & identifieras med romerska siffror, RNR här...]

Specialfall:
============
* Uppslag - kan vara en blandning av de flesta redan
uppräknade typerna, t.ex 'ucint + prel Ir' eller '78v + prel IIr'
* Originalpaginering:	ersätter 'f' med 'p'
* Detaljer - även ett uppslag kan ha detaljer...:	-detail


Så, med hänsyn till ovanstående, bör följande exempel fungera:

Volym 98, prel 2 verso + folio 1 recto:
  Mh_98_f_II_v-f_1_r

Volym 24, detalj av främre pärmens utsida:
  Mh_24_ucext-detail_2

Volym 23, folio 231 recto:
  Mh_23_f_321a_r
  

Utsidan använder de ovan givna 8 namnen.
En sida definieras som : (f|p)_(NR|RNR)(\w)_(r|v)
Den extra bokstaven (\w) beror på sidor som '432b' & sånt.

Ett uppslag är en kombination av antingen 2 sidor, eller 1 sida
och 'ucint' || 'lcint'. Kombinationen separeras med '-'.

Detaljer är ett möjligt tillägg till sidor/utsida/uppslag, anges
som suffix: -detail_<NR>.
